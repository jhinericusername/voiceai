# Ashby Self-Serve Onboarding Design

## Summary

Puddle will replace the current ops-assisted Ashby setup with a customer-facing
company onboarding flow. One authorized Puddle user from a company connects the
company's Ashby workspace by entering an Ashby API key, selecting jobs, and
manually configuring Ashby webhooks with values generated by Puddle. After the
webhook `ping` succeeds, teammates from the same company domain or WorkOS
organization bypass Ashby setup and can use the shared integration.

This is a v1 direct Ashby integration. It intentionally avoids Ashby OAuth,
marketplace setup, and a generalized ATS abstraction.

## Current Context

- The backend already owns Ashby integration state in RDS through
  `ashby_company_integrations`, `ashby_applications`,
  `ashby_webhook_events`, and `ashby_candidate_scores`.
- The platform dashboard currently asks the backend for company state and shows
  either recent screens or an internal Ashby setup panel.
- Candidate search and score saving already flow through authenticated platform
  API routes into backend Ashby routes.
- The current internal setup route requires operators to call
  `/integrations/ashby/setup` directly with an API key and selected job IDs.
- The current platform webhook route verifies all Ashby webhooks with one global
  `PUDDLE_ASHBY_WEBHOOK_SECRET`, then forwards parsed JSON to the backend.
- The infra stack already has a Secrets Manager pattern for runtime secrets and
  ECS secret injection.

## Goals

1. Let a customer admin complete Ashby setup from the Puddle dashboard.
2. Store customer Ashby API keys and webhook secrets safely without exposing
   them to the browser or platform container.
3. Keep platform-owned runtime secrets in AWS Secrets Manager.
4. Keep customer integration state queryable in RDS for same-domain bypass,
   setup status, selected jobs, sync state, and support/debugging.
5. Use manual Ashby webhook setup for v1: Puddle displays the request URL,
   generated secret token, and required event checklist.
6. Move Ashby webhook signature verification into the backend so verification
   can use the correct per-company webhook secret.
7. Preserve the current score tab behavior: scores are saved in Puddle only and
   are not written back to Ashby in v1.

## Non-Goals

- No Ashby OAuth or marketplace installation.
- No automatic webhook creation in v1, even though Ashby exposes a
  `webhook.create` endpoint, because that requires broader API key permissions.
- No customer-managed AWS secrets.
- No one-Secrets-Manager-secret-per-customer design in v1.
- No score writeback to Ashby.
- No multi-ATS abstraction.

## Secret Storage Decision

Use encrypted RDS fields for customer-provided Ashby secrets, protected by one
backend-only platform encryption key stored in AWS Secrets Manager.

Infra creates or imports:

```text
/puddle-videoagent/integrations/encryption-key
```

The backend service and backend migration task receive this value as:

```text
PUDDLE_INTEGRATION_SECRET_KEY
```

The platform service does not receive this key. The browser never receives this
key. The only runtime that can decrypt customer Ashby credentials is the backend.

This is the preferred v1 tradeoff because it fits the existing RDS-backed
integration model, keeps company lookup simple, avoids a secret lifecycle per
customer, and still keeps customer credentials encrypted at rest at the
application layer in addition to RDS storage encryption.

## Data Model

Add a migration that evolves `ashby_company_integrations` from internal setup to
self-serve onboarding.

Rows are created only after the backend validates and encrypts an Ashby API key.
Before that point, company state returns an API key setup state without storing
an integration row.

Required columns:

```text
ashby_company_integrations
- integration_id text primary key
- organization_id text null
- email_domain text not null
- ashby_api_key_ciphertext text not null
- ashby_webhook_secret_ciphertext text not null
- selected_job_ids text[] not null default '{}'
- setup_status text not null default 'pending_webhook'
- connected_at timestamptz null
- last_ping_at timestamptz null
- last_sync_at timestamptz null
- created_by_email text null
- updated_by_email text null
- created_at timestamptz not null default now()
- updated_at timestamptz not null default now()
```

`setup_status` values:

- `missing`: no row exists. This is represented in API responses, not stored.
- `api_key_pending`: no valid Ashby API key has been submitted. This is
  represented in API responses before row creation, not stored in
  `setup_status`.
- `job_selection_pending`: the API key is valid but no jobs have been selected.
- `pending_webhook`: jobs are selected and Puddle is waiting for Ashby `ping`.
- `connected`: a valid signed `ping` was processed.
- `error`: setup failed in a recoverable way and the UI should show retry state.

The existing unique indexes on `email_domain` and non-null `organization_id`
remain. Existing rows created by the internal route should be backfilled with a
generated encrypted webhook secret and a status based on `connected_at`.

## Backend API

The backend owns all secret handling and Ashby calls.

### `POST /integrations/ashby/company-state`

Input: company identity from the platform.

Response includes:

```json
{
  "connected": false,
  "setupStatus": "pending_webhook",
  "integrationId": "int_...",
  "emailDomain": "company.com",
  "selectedJobIds": ["job_..."],
  "lastPingAt": null,
  "lastSyncAt": null,
  "webhookUrlPath": "/api/ashby/webhook?integrationId=int_..."
}
```

The response must not include decrypted secrets by default.

### `POST /integrations/ashby/onboarding/api-key`

Input:

```json
{
  "emailDomain": "company.com",
  "organizationId": "org_...",
  "reviewerEmail": "admin@company.com",
  "ashbyApiKey": "..."
}
```

Behavior:

1. Validate company identity.
2. Validate the API key by calling Ashby's `job.list` endpoint, because the
   next onboarding step needs the accessible jobs.
3. Encrypt and store the API key.
4. Generate and encrypt a per-company webhook secret if one does not already
   exist.
5. Return available jobs for selection.

### `POST /integrations/ashby/onboarding/jobs`

Input:

```json
{
  "emailDomain": "company.com",
  "organizationId": "org_...",
  "reviewerEmail": "admin@company.com",
  "selectedJobIds": ["job_..."]
}
```

Behavior:

1. Require an existing integration with a valid encrypted API key.
2. Store selected job IDs.
3. Return the webhook setup payload.

Response includes the generated webhook secret exactly when needed for setup:

```json
{
  "integrationId": "int_...",
  "webhookUrl": "https://app.usepuddle.com/api/ashby/webhook?integrationId=int_...",
  "webhookSecret": "...",
  "requiredEvents": [
    "ping",
    "applicationSubmit",
    "applicationUpdate",
    "candidateStageChange",
    "candidateDelete",
    "candidateMerge",
    "candidateHire"
  ]
}
```

The backend may return the decrypted webhook secret during onboarding because
the setup user must paste it into Ashby. The platform should render it as a
sensitive copy field and never log it.

### `POST /integrations/ashby/onboarding/sync`

Behavior:

1. Require an integration with API key and selected jobs.
2. Decrypt the API key.
3. Run active application sync for selected jobs.
4. Store `last_sync_at`.
5. Return `syncedCount`.

This can be called immediately after job selection and again after webhook ping.

### `POST /integrations/ashby/webhook`

Input from the platform webhook proxy:

```json
{
  "integrationId": "int_...",
  "companyDomain": null,
  "rawBody": "{...}",
  "signature": "sha256=..."
}
```

Behavior:

1. Resolve the integration by `integrationId` first, then by `companyDomain`
   only as a fallback for existing internal URLs.
2. Decrypt `ashby_webhook_secret_ciphertext`.
3. Verify the `Ashby-Signature` digest against the raw body.
4. Parse JSON only after signature verification succeeds.
5. Process `ping` by setting `connected_at`, `last_ping_at`, and
   `setup_status = 'connected'`.
6. Deduplicate non-ping events by `webhookActionId`.
7. Upsert or deactivate synced applications based on webhook action.

Invalid signatures return `401` and are not processed.

## Platform Flow

The dashboard becomes a setup wizard only when the company is not connected.

Step 1: API key

- Explain that an Ashby admin should paste an API key.
- Submit to `/api/ashby/onboarding/api-key`.
- Show validation errors without storing anything in browser state beyond the
  current form.

Step 2: job selection

- Show jobs returned by the backend.
- Let the user select one or more active jobs.
- Submit to `/api/ashby/onboarding/jobs`.

Step 3: webhook setup

- Show the generated webhook URL.
- Show the generated webhook secret as a copyable sensitive value.
- Show the required event checklist.
- Show the current status: waiting for `ping`, connected, or error.

Step 4: sync and finish

- Let the user run initial sync.
- Once `connected_at` exists, the dashboard shows recent screens.
- Same-domain teammates see the connected state and do not see setup.

The platform adds authenticated API routes that derive company identity from
WorkOS and forward onboarding requests to the backend:

```text
POST /api/ashby/onboarding/api-key
POST /api/ashby/onboarding/jobs
POST /api/ashby/onboarding/sync
```

The existing unauthenticated webhook route changes to preserve the raw body and
forward it with `Ashby-Signature` to the backend. It no longer verifies a global
platform webhook secret.

## Infra Changes

Extend the existing `RuntimeSecrets` shape with:

```text
integrationEncryptionKey: integrations/encryption-key
```

Create it with the existing `createSecret` helper. Grant read access only to the
backend ECS execution role. Inject it into:

- backend service container as `PUDDLE_INTEGRATION_SECRET_KEY`
- backend migration container as `PUDDLE_INTEGRATION_SECRET_KEY`

Do not inject it into:

- platform container
- agent container
- browser-visible `NEXT_PUBLIC_*` variables

Remove the need for `PUDDLE_ASHBY_WEBHOOK_SECRET` from platform runtime config
after backend webhook verification is implemented.

## Security Requirements

- Never send the Ashby API key back to the platform after submission.
- Never log the Ashby API key, decrypted webhook secret, raw webhook body, or
  Ashby signature.
- Only return the decrypted webhook secret during setup actions that require
  customer copy/paste.
- Store encrypted values with authenticated encryption.
- Keep the encryption key out of RDS.
- Keep the encryption key out of the platform task.
- Validate selected job IDs against jobs returned from Ashby for that API key.
- Same-domain bypass does not bypass Puddle authentication or allowed-domain
  checks.
- Any reset/reconnect action must overwrite the encrypted API key and generate
  or intentionally preserve the webhook secret based on explicit backend logic.

## Error Handling

- Invalid API key returns a clear setup error and does not mark the integration
  connected.
- API key missing required read permissions returns a permissions-focused error.
- Failed job fetch keeps the user on the API key step.
- Empty job selection returns `400`.
- Webhook ping with invalid signature returns `401`.
- Webhook ping for an unknown integration returns `404`.
- Duplicate webhook events return `200` when already processed.
- Sync failures keep the integration in setup state and show retry guidance.

## Testing

Backend unit tests:

- Encryption still round-trips and rejects the wrong key.
- API key onboarding validates company identity.
- API key onboarding encrypts and stores the API key.
- Job selection stores selected jobs and returns webhook setup values.
- Company state never returns API key plaintext.
- Webhook route rejects invalid signatures before parsing JSON.
- Webhook route verifies signatures with the per-company webhook secret.
- Ping updates `connected_at`, `last_ping_at`, and `setup_status`.
- Initial sync uses the decrypted API key and selected jobs.

Platform tests:

- Onboarding API routes require WorkOS auth.
- Onboarding API routes reject disallowed email domains.
- Onboarding API routes forward derived company identity rather than trusting
  client-provided identity.
- Webhook route forwards raw body and signature to the backend.
- Dashboard shows setup steps when not connected.
- Dashboard shows recent screens when connected.

Infra tests:

- Secrets Manager resource count increases by one.
- Backend service and migration task receive
  `PUDDLE_INTEGRATION_SECRET_KEY`.
- Platform and agent tasks do not receive `PUDDLE_INTEGRATION_SECRET_KEY`.
- Backend execution role can read the integration encryption key.
- Platform execution role cannot read the integration encryption key.

Manual verification:

1. Deploy dev with migrations.
2. Sign in to `/dashboard` as an allowed company user.
3. Paste an Ashby API key.
4. Select one or more Ashby jobs.
5. Copy webhook URL and secret into Ashby for required events.
6. Confirm Ashby `ping` marks the integration connected.
7. Run initial sync.
8. Sign in as another same-domain user and confirm setup is bypassed.
9. Score an active Ashby candidate and confirm the score appears on
   `/dashboard`.

## Rollout

1. Add infra secret and ECS injection.
2. Add migration for webhook secret and setup status.
3. Add backend onboarding endpoints.
4. Move webhook signature verification to backend.
5. Add platform onboarding API routes.
6. Replace the internal setup panel with the setup wizard.
7. Update the Ashby setup runbook for dev/prod manual verification.

The existing internal setup route can remain temporarily as an operator-only
compatibility path, but the dashboard should use the new onboarding endpoints.
